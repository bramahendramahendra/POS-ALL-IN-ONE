# Prompt 6.1 — Deteksi Konflik di Backend

## File Diubah
`backend/domain/sync/service/sync_service.go`, `backend/domain/sync/repo/sync_repo.go`

## Konsep
Saat desktop mengirim data offline ke backend (push sync), backend membandingkan timestamp data desktop vs data yang sudah ada di MySQL. Jika ada perbedaan, data ditandai sebagai konflik dan dimasukkan ke tabel `sync_conflicts`.

## Logika Deteksi Konflik

```go
// sync_service.go
func (s *syncService) DetectConflict(desktopData SyncItem) (bool, error) {
    // Ambil data yang sama dari MySQL (by entity + server_id)
    existing, err := s.repo.GetByEntityAndID(desktopData.Entity, desktopData.ServerID)
    if err != nil || existing == nil {
        return false, nil // tidak ada data online → tidak ada konflik, langsung apply
    }

    // Bandingkan updated_at — jika data online lebih baru dari data desktop
    onlineTime, _ := time.Parse(time.RFC3339, existing.UpdatedAt)
    desktopTime, _ := time.Parse(time.RFC3339, desktopData.UpdatedAt)

    if onlineTime.After(desktopTime) {
        // Konflik: data online sudah diubah setelah desktop terakhir sync
        return true, nil
    }

    return false, nil
}
```

## Tabel sync_conflicts (MySQL)

```sql
CREATE TABLE sync_conflicts (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    entity          VARCHAR(50) NOT NULL,    -- 'transaction' | 'expense' | dll
    entity_id       BIGINT,                  -- ID di server
    local_id        VARCHAR(36),             -- UUID lokal dari desktop
    device_id       VARCHAR(100) NOT NULL,   -- identifier device desktop
    desktop_data    JSON NOT NULL,           -- snapshot data versi desktop
    online_data     JSON NOT NULL,           -- snapshot data versi online
    status          ENUM('pending','resolved') DEFAULT 'pending',
    resolved_action ENUM('approve','reject'),
    resolved_by     BIGINT,                  -- user_id yang resolve
    resolved_at     DATETIME,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_status (status),
    INDEX idx_entity (entity, entity_id)
);
```

## Push Sync Handler

```go
// handler: POST /sync/push
func (h *SyncHandler) PushSync(c *gin.Context) {
    var items []dto.SyncItemRequest
    c.BindJSON(&items)

    results := []dto.SyncItemResult{}
    for _, item := range items {
        hasConflict, _ := h.service.DetectConflict(item)
        if hasConflict {
            // Simpan ke sync_conflicts, jangan apply ke MySQL dulu
            conflictID, _ := h.service.SaveConflict(item)
            results = append(results, dto.SyncItemResult{
                LocalID: item.LocalID,
                Status:  "conflict",
                ConflictID: conflictID,
            })
        } else {
            // Tidak ada konflik — apply langsung
            serverID, _ := h.service.ApplyItem(item)
            results = append(results, dto.SyncItemResult{
                LocalID:  item.LocalID,
                Status:   "synced",
                ServerID: serverID,
            })
        }
    }

    c.JSON(200, gin.H{"results": results})
}
```

## Output yang Diharapkan
Backend dapat mendeteksi konflik antara data desktop vs online saat proses push sync, dan menyimpan konflik ke tabel `sync_conflicts`.
